# Monitoring & Alerts

## Overview

Every component emits Prometheus metrics and structured logs. Grafana visualizes the data. Discord webhooks provide real-time alerts across dedicated channels.

## Prometheus Metrics

All metrics are exposed via `prometheus-client` on a `/metrics` endpoint scraped by Prometheus.

### Counters

| Metric | Type | Labels | Description |
|---|---|---|---|
| `trading_orders_submitted_total` | Counter | broker, symbol, side | Total orders submitted |
| `trading_orders_filled_total` | Counter | broker, symbol, side | Total orders filled |
| `trading_orders_failed_total` | Counter | broker, reason | Total orders that failed |
| `trading_risk_blocks_total` | Counter | rule, strategy | Total signals blocked by risk manager |
| `trading_signals_generated_total` | Counter | strategy, symbol | Total signals generated by strategies |
| `trading_events_published_total` | Counter | event_type | Total events published to bus |
| `trading_events_consumed_total` | Counter | event_type, consumer | Total events consumed from bus |

### Gauges

| Metric | Type | Labels | Description |
|---|---|---|---|
| `trading_portfolio_value` | Gauge | broker | Current portfolio value |
| `trading_open_positions` | Gauge | broker, symbol | Number of open positions |
| `trading_drawdown_pct` | Gauge | broker | Current drawdown percentage |
| `trading_exposure_pct` | Gauge | broker, symbol | Current exposure per symbol |
| `trading_websocket_lag_seconds` | Gauge | broker | Seconds since last WS message |

### Histograms

| Metric | Type | Labels | Description |
|---|---|---|---|
| `trading_order_latency_seconds` | Histogram | broker | Time from signal to order submission |
| `trading_fill_latency_seconds` | Histogram | broker | Time from order submission to fill |
| `trading_strategy_eval_seconds` | Histogram | strategy | Time for strategy evaluation |
| `trading_sentiment_eval_seconds` | Histogram | model | Time for LLM sentiment evaluation |

## Grafana Dashboards

### Dashboard 1: Trading Overview

**Purpose:** Live view of portfolio performance, positions, and order flow.

Panels:
- Portfolio value over time (line chart)
- Cumulative PnL (line chart)
- Current open positions by symbol (table)
- Orders submitted vs filled vs failed (stacked bar chart, 1h buckets)
- Drawdown percentage over time (line chart with threshold line at max allowed)
- WebSocket connection status and lag (stat panel)

### Dashboard 2: Risk Metrics

**Purpose:** Real-time risk monitoring and blocked trade analysis.

Panels:
- Current drawdown % vs maximum allowed (gauge with red zone)
- Exposure per symbol as % of portfolio (bar chart)
- Correlation matrix of open positions (heatmap)
- Risk blocks over time by rule type (stacked area chart)
- Daily trade count vs maximum allowed (stat panel)
- Top 10 blocked signals by reason (table)

### Dashboard 3: Sentiment (Phase 3+)

**Purpose:** Monitor LLM sentiment pipeline health and output.

Panels:
- Sentiment score distribution (histogram)
- Sentiment by source (news vs social) over time (line chart)
- LLM evaluation latency (line chart)
- Confidence score distribution (histogram)
- Sentiment accuracy vs price movement (scatter plot, retrospective)

### Dashboard 4: System Health

**Purpose:** Infrastructure monitoring.

Panels:
- Redis connection status and Pub/Sub message rate (stat + line)
- TimescaleDB connection pool usage (line chart)
- Event bus throughput (messages/second by topic)
- Process memory and CPU (line chart)
- GC pause spikes (line chart — watcher for latency jitter)

## Discord Alert Channels

All alerts go to a Discord server with dedicated channels. Each channel has a specific purpose and severity level.

### Channel: `#trading-signals`

- Every signal generated by a strategy
- Includes: symbol, side, strategy name, confidence score
- Low noise, informational
- Example: `🟢 SIGNAL | BUY BTC/USDT | Strategy: sma_crossover | Confidence: 0.78`

### Channel: `#trading-executions`

- Every order submitted, filled, or failed
- Includes: symbol, side, quantity, price, broker, fill time
- Example: `✅ FILL | BUY 0.01 BTC/USDT @ $67,234 | Binance Live | Slippage: +0.02%`
- Example: `❌ FAIL | SELL ETH/USDT | Binance Live | Reason: Insufficient balance`

### Channel: `#risk-alerts`

- Every signal blocked by the risk manager
- Every drawdown threshold breach
- Every exposure limit breach
- Example: `⛔ RISK BLOCK | BUY BTC/USDT | Rule: MaxExposure | Reason: BTC already 4.8% of portfolio (max 5%)`
- Example: `🔴 DRAWDOWN ALERT | Current drawdown: -3.2% | Max allowed: -3%`

### Channel: `#system-errors`

- WebSocket disconnections and reconnections
- Broker API errors (5xx, 429 rate limits)
- Dead-letter queue entries
- Process restarts
- Stale data detection
- Example: `⚠️ WS DISCONNECT | Binance | Reconnecting...`
- Example: `🔴 STALE DATA | No bar received for BTC/USDT in 120 seconds`

### Channel: `#sentiment` (Phase 3+)

- Notable sentiment shifts (score change > 0.3 in 1 hour)
- LLM pipeline errors (model unavailable, malformed output)
- Source scraper failures
- Example: `📊 SENTIMENT SHIFT | BTC | Score: 0.82 → 0.21 | Source: crypto_news | Confidence: 0.71`

## Alert Severity Levels

| Level | Color | Action | Channels |
|---|---|---|---|
| Info | Blue | Log only, no action needed | `#trading-signals`, `#trading-executions` |
| Warning | Yellow | Monitor, may need attention | `#risk-alerts`, `#system-errors` |
| Critical | Red | Immediate investigation required | `#risk-alerts`, `#system-errors`, ping @here |
| Emergency | Red + Ping | System stopped trading, manual intervention | `#system-errors`, ping @everyone |

## Docker Compose Services

```yaml
services:
  redis:
    image: redis:7-alpine
    ports: ["6379:6379"]
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s

  timescaledb:
    image: timescale/timescaledb:latest-pg16
    ports: ["5432:5432"]
    environment:
      POSTGRES_DB: trading
      POSTGRES_PASSWORD: ${DB_PASSWORD}
    volumes: ["timescaledb_data:/var/lib/postgresql/data"]
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s

  grafana:
    image: grafana/grafana:latest
    ports: ["3000:3000"]
    volumes:
      - ./grafana/provisioning:/etc/grafana/provisioning
      - grafana_data:/var/lib/grafana

  prometheus:
    image: prom/prometheus:latest
    ports: ["9090:9090"]
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml

volumes:
  timescaledb_data:
  grafana_data:
```
